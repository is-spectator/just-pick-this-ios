# Recommendation Card Feedback Routes Report

Date: 2026-06-23

## Issue

Addresses a backend slice of ISS-017 Feedback UX and ISS-028 Post-experience Review from `pipi_effect_iteration_issues.xlsx`.

Before this slice, accepting a recommendation card had a dedicated endpoint, but rejection/change feedback required callers to construct generic `/v1/events` payloads by hand.

## What Changed

- Added `POST /v1/cards/{card_id}/reject`.
- Added `POST /v1/cards/{card_id}/change`.
- Both endpoints:
  - validate the card exists
  - update the card status (`rejected` / `changed`)
  - write a `UserBehaviorEvent`
  - feed existing intent-memory rejection counters
  - feed existing user preference memory negative signals

## Non-goals

- No new recommendation is generated by `change`.
- No iOS UI was changed.
- No agent routing or PipiLoop behavior was changed.
- No database migration was needed.

## Tests

- `app/tests/test_user_behavior_events.py::test_card_reject_and_change_routes_write_feedback_events`

## Follow-ups

- The mobile UI can now call dedicated endpoints for "不喜欢 / 换一个" instead of emitting generic events.
- A later slice can expose richer feedback tags from the UI without changing the behavior event ledger.
