下一步优化方向

1

路由质量：先追问、再求助、最后推荐

当前最容易错的是把泛需求直接变成 help_card，或把店内点菜误判成区域选店。优先建立更多 product-path 回归用例。

2

证据质量：把“可推荐”变成可解释评分

Evaluator 应输出 can_recommend、confidence、missing_requirements，并写进 AgentRun，运营后台可直接看。

3

LLM 接入：只从 Shadow Mode 开始

线上仍走 deterministic；LLM 输出 ReasonerDecision schema，记录 mismatch，再用 benchmark 判断是否升主路径。

4

工具面收窄：减少误选工具

AbilityCenter 按 InputGate.allowed_tools 授权，publish/update/finalize 不能被普通推荐请求调用。

5

自进化闭环：报告生成 issue

benchmark 自动产出 seed gap、agent issue、low quality case，下一轮只修报告里真实失败。

2. 第一层路由分支

Iteration 1 的核心是把“该追问”和“该求助”拆开。求助卡不再是默认兜底，unknown 默认追问。

Branch A

Chitchat / App Help

例：你好、你是谁、谢谢、怎么用。

  • `should_enter_loop=false`
  • 不建 Question
  • 不建 RetrievalRun / ToolCall
  • `ui_events=[]`
  • 直接 assistant text
Branch B

Clarification

例：我想吃饭、帮我选一家、帮我点菜、附近有啥。

  • `response_kind=clarification`
  • `missing_slots` 写入 metadata
  • 不创建推荐卡
  • 不创建求助卡
  • 问题要明确补哪个槽位
Branch C

Venue Ordering

例：三里屯海底捞，两个人不太能吃辣，帮我点。

  • venue + ordering 优先于 area
  • `location_state=in_venue`
  • `target_type=ordering_bundle`
  • 先 search，再由 tool 创建 card
  • 无 seed 时求助卡保留店内上下文
Branch D

Area Food

例：北京市朝阳区热干面、三里屯川菜、望京 SOHO 清淡。

  • `location_state=in_area`
  • 允许 AMap POI / local seed / intent answer
  • 无足够 evidence 可结构化 help_card
  • 不得泛化成“北京这顿饭，求一个”
  • debug 可见 canonical_query 和 slots

3. PipiLoop 内循环

PipiLoop 是唯一单轮 agent engine。Reasoner 每轮只能输出 `tool` 或 `answer`，工具结果必须回灌后才能进入下一轮。

阶段 输入 输出 / 约束
Reasoner context_pack + tool_results 只能输出 ToolDecision 或 AnswerDecision。
AbilityCenter tool_name + tool_args 校验 allowed_tools、schema、权限；写 ToolCall。
Tool Search / Card / Help / Publish / AMap 真正写 DB；推荐卡和求助卡不能绕过 tool。
Evaluator ToolResult + state 打分，低质时建议 retry / help / safe answer。
AnswerGate AnswerDecision 校验 ui_events 的 card_id/help_card_id 已存在。

4. Trace、Admin 与自进化数据流

每次请求都要能从用户看到的结果，反查到 route、context、tool、evidence、evaluator 和 answer gate。这样测试报告才能变成下一轮工程任务。

Trace 必含事件

写在 AgentRun.output_json.loop_trace。

  • input_gate_result
  • context_pack
  • reasoner_decision
  • tool_call
  • tool_result
  • evaluator_result
  • answer_gate_result

Admin 需要看什么

运营后台不只是查 DB，而是查“为什么这轮这么答”。

  • session / turn / trace 明细
  • query rewrite 和 extracted slots
  • selected_tool 与失败原因
  • retrieval hits / AMap run
  • shadow LLM mismatch

Benchmark 反推修复

每个失败 case 都要能归类，不再笼统说“模型不行”。

  • route gap：InputGate / rewrite
  • seed gap：缺 approved answer
  • evidence gap：证据弱
  • tool gap：schema / permission
  • copy gap：文案和卡片质量

一句话版本

皮皮的可优化点不在“多写几个 if/else”，而在每一轮都能被 gate 约束、被 tool 执行、被 evaluator 打分、被 trace 回放、被 benchmark 反推下一轮修复。

文件用途：内部架构 review / 运营后台讨论 / 评测修复排期。此页面不接入用户端，不依赖移动端设计稿。
